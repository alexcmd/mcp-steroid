## 0.96

### IntelliJ Platform 261 (2026.1) becomes the primary build target; 262 EAP added as a secondary verification target

The plugin no longer builds or runs against IntelliJ 2025.3 (`253`). The build target is now `261` (IntelliJ IDEA 2026.1) — `sinceBuild = "261"`, `MANAGED_BACKEND_MIN_SUPPORTED_BUILD = "261"`, root Kotlin 2.3.20 (matches IDE 261's bundled `kotlin-stdlib` in `lib/util-8.jar`, verified by reading bytecode constants of `kotlin/KotlinVersion.class`). Plugin Verifier runs against both 261 (`261.22158.277`) and `262-EAP-SNAPSHOT` (`262.6228.19`) — **0 internal-API usages** against either.

### kotlinx libraries pinned to what 261 bundles; never shipped in the plugin .zip

`kotlinx-coroutines-core` pinned to `1.10.2`, `kotlinx-serialization-{core,json}` pinned to `1.9.0` — the closest public Maven Central versions to what IntelliJ 261 bundles inside `lib/intellij.libraries.kotlinx.*.jar`. The existing `configurations.implementation.exclude(group = "org.jetbrains.kotlinx")` in `ij-plugin/build.gradle.kts` continues to keep these jars OUT of the plugin .zip — at runtime the plugin reuses the IDE-bundled copies (per [JetBrains' coroutines-libraries rule](https://plugins.jetbrains.com/docs/intellij/using-kotlin.html#coroutinesLibraries)).

Two new gates fail the build fast if a compile-time pin drifts away from what the IDE actually ships:

- `:ij-plugin:verifyBundledLibraries` — manifest-side equality check against the IDE's `lib/intellij.libraries.kotlinx.*.jar` JAR manifests.
- `:ij-plugin:verifyBundledKotlinxRuntime` — runtime classloader probe (`intellij-downloader`'s `KotlinxRuntimeProbe`) that exercises `Json.encodeToString` / `decodeFromString` / `parseToJsonElement` / `buildJsonObject` plus coroutines `runBlocking` / `withContext(Dispatchers.Default)` / `Channel.send-receive` / `MutableStateFlow.value` / `Flow.fold` / `CompletableDeferred.await` against a classpath of ONLY the IDE's `lib/*.jar`. Either a `LinkageError` or a `NoClassDefFoundError` at link time fails the build.

### `intellij-downloader` — in-repo IDE artifact resolution; no more `useInstaller`

A new in-repo Gradle subproject resolves IntelliJ IDE archives, unpacks them to `build/local-ides/IU-<full-build>-<os>-<arch>/`, and exposes their root dirs to the IntelliJ Platform Gradle Plugin via `local(file)` selectors. `ij-plugin/build.gradle.kts` no longer uses `useInstaller = true`; `intellijIdeaUltimate(...)` and `pluginVerification.ides { }` both go through `ideRootProviderFor(...)`. Named per-major EAP tags like `262-EAP-SNAPSHOT` resolve through the products API (the IPGP `nightly()` Maven layout uses the bare `262-SNAPSHOT` form). The unpacker writes a `.mcp-steroid-unpack-complete` marker under a `FileLock` so concurrent Gradle workers don't trip over each other.

### Gradle daemon + every module: JDK 21 → JDK 25

`gradle/gradle-daemon-jvm.properties` pins `toolchainVersion=25`. Every `build.gradle.kts` declares `kotlin { jvmToolchain(25) }`. The GitHub Actions build image (`docker/build/Dockerfile`) and the four CLI integration-test Docker images (`test-helper/src/main/docker/{mcp,claude,codex,gemini}-cli/Dockerfile`) all install `temurin-25-jdk` / `temurin-25-jre`. The bump matches IntelliJ 261's bundled JBR.

### Test execution hardened: all host-side test JVMs run headless

Root `build.gradle.kts` now sets `systemProperty("java.awt.headless", "true")` on every `Test` task across every subproject. The only non-headless surface is the Docker IDE containers under `:test-integration` / `:test-experiments` where the IDE actually renders. Previously `:ij-plugin:test` could pop a non-modal `JFrame` from a host-side execution test — that test was deleted (it tested behavior the headless guarantee now makes vacuous; the modal-DialogWrapper path is covered by `DialogKillerIntegrationTest` in Docker + Xvfb).

### Fixed Issues

- `:ij-plugin:verifyBundledLibraries` / `:verifyBundledKotlinxRuntime` close the long-standing compile-time-vs-runtime drift gap that previously could only surface at IDE-install time.

### Plugin Verifier

Compatible against IntelliJ 2026.1 (`IU-261.22158.277`) and 2026.2 EAP (`IU-262.6228.19`). **0 internal-API usages** against either (hard gate). Experimental API usages unchanged from 0.95.0.
