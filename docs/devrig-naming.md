# devrig naming — projects & IDEs

This is the canonical spec for how the **devrig** binary names projects
and IDEs in every output it produces (CLI text, CLI JSON, devrig stdio
MCP).

## Scope

devrig is an **independent binary**. Its output is consumed by:

1. **Humans** reading `devrig backend` / `devrig project` text.
2. **Scripts** consuming `devrig backend --json` / `devrig project --json`.
3. **AI agents** calling `steroid_list_projects` / `steroid_list_windows`
   over the devrig stdio MCP server (`devrig mpc`).

All three surfaces share the names defined here. **They are not meant to
be valid against the in-IDE MCP Steroid server** — that's a separate
endpoint with its own (legacy) naming. devrig is free to redefine.

## Vocabulary

| Term | Meaning |
|---|---|
| **original name** | The unmodified label as reported by the source. For a project: `Project.name` from the running IntelliJ. For an IDE/backend: `"<IdeInfo.name> <IdeInfo.version>"` (marker), `productFullName` (port), `"<displayName> <version>"` (available download), or `"<productKey> <version>"` (managed). |
| **exposed name** | The canonical id string exposed by devrig. It is the only id this spec defines for a row. Always `<slug(originalName)>-<hash8>`, where `slug(s)` is defined below. devrig always passes the exposed name as one argv token; consumers must not split it on whitespace. |
| **hash8** | First 6 bytes of `SHA-256` over the canonical inputs, base64url-encoded without padding (8 chars, URL-safe). |
| **bootHash** | Long base64url string (≥ 22 chars, i.e. ≥ 16 random bytes encoded without padding) generated **once per JVM start** by the MCP Steroid plugin and held in an `@Service(Service.Level.APP)`. Same value across the plugin process's entire lifetime. Written verbatim into every marker refresh. **Never regenerated by the marker writer** — project open / close, plugin reload of unrelated state, and explicit marker rewrites all reuse the existing value. A fresh `bootHash` only appears when the IDE JVM restarts (or the plugin is uninstalled+reinstalled within the same JVM, which is treated as "plugin startup"). Published in the pid marker and on the plugin's identity API. |
| **archiveSha256** | Hex-encoded SHA-256 of the IDE archive that `devrig backend download <id>` would fetch. **Nullable.** When present, the JetBrains products API published a `checksumLink` for this `(productKey, version, os, arch)` tuple and devrig resolved it to the canonical SHA-256. When `null`, the products API did not publish a checksum link (older EAP rows, some legacy stable releases, transient feed gaps). Listing **does not** synchronously fetch checksums; the value is whatever the feed already exposes. |
| **startedTime** | ISO-8601 instant captured when the plugin generated its `bootHash`. Published in the same marker payload. Part of the JSON data model; **not** folded into `hash8` today. |
| **locator label** | The parenthesised "how do I reach this" hint, separate from the name. E.g. `(build IU-261…, pid 24017)`. |

### Slug rule

`slug(s)` is the function that converts an `originalName` into the
argv-token form used in exposed names. It MUST satisfy:

- Replace every maximal run of bytes in the set
  `{0x09, 0x0A, 0x0B, 0x0C, 0x0D, 0x20}` (HT, LF, VT, FF, CR,
  SPACE — i.e. `Character.isWhitespace`'s ASCII subset) with a
  single `_`. Non-ASCII whitespace code points (NBSP, ideographic
  space, etc.) are intentionally preserved — they survive into the
  exposed `id`, which is still a single argv token and round-trips
  through `ProcessBuilder` byte-for-byte.
- Leave every other code point unchanged. In particular, do not
  lowercase, do not strip punctuation, do not normalise Unicode.
- Is idempotent: `slug(slug(s)) == slug(s)`.
- Total: defined for every input string including empty.

`originalName` continues to carry the unmodified label; only the
composed exposed name uses the slugged form. Consumers that need the
human-readable label use `originalName`; consumers that need a
routing key use `id`.

devrig treats the slugged exposed name as a single argv token. It is
NOT necessarily a safe `sh -c` token — `slug()` deliberately
preserves all non-whitespace code points to keep `originalName`
recognisable in the rendered id. Consumers execute via
`actions[].argv` (which is shell-agnostic) rather than
`actions[].command` (which is for display only).

Examples:

| originalName | slug |
|---|---|
| `IntelliJ IDEA 2025.3.3` | `IntelliJ_IDEA_2025.3.3` |
| `IntelliJ IDEA Ultimate` | `IntelliJ_IDEA_Ultimate` |
| `idea-community 2025.3.3` | `idea-community_2025.3.3` |
| `myproject` | `myproject` |

### `bootHash` lifecycle

The IDE-side plugin owns `bootHash` generation. Implementation
contract (binding on every plugin release):

1. The plugin defines an application-level service:

   ```kotlin
   @Service(Service.Level.APP)
   class McpSteroidBootHash {
       val value: String = generateBootHash()  // call once, cached
       val startedTime: Instant = Instant.now()

       private fun generateBootHash(): String {
           val bytes = ByteArray(16)
           SecureRandom().nextBytes(bytes)
           return Base64.getUrlEncoder().withoutPadding().encodeToString(bytes)
       }
   }
   ```

2. `bootHash` is computed inside the service constructor — i.e. on
   the **first** call to `service<McpSteroidBootHash>()`. Subsequent
   calls return the cached value.

3. Every marker writer call — `ServerUrlWriter`, or any future
   writer — reads `service<McpSteroidBootHash>().value` /
   `.startedTime` and inlines them into the marker payload. The
   writer MUST NOT call `generateBootHash` directly and MUST NOT
   construct a fresh `McpSteroidBootHash` instance.

4. Plugin hot-reload (IntelliJ's dynamic plugin reload during
   development) creates a fresh service instance and therefore a
   fresh `bootHash`. This is treated the same as a JVM restart —
   exposed names flip.

5. `startedTime` is the moment the service was first instantiated.
   It is a published data-model field; not folded into the hash
   today.

The contract above is what callers see; the implementation may
differ in details (e.g. lazy `by lazy { ... }`, alternative
randomness source) as long as the **invariant holds**: within one
plugin process, every marker refresh carries the same `bootHash`,
and a fresh `bootHash` only appears when the plugin process
restarts (or hot-reload triggers re-instantiation).

### The only ID

Every consumer-visible reference to a project or a backend in any
devrig output uses the **exposed name** — `<slug(originalName)>-<hash8>`
— and nothing else. There are no alias IDs, no historical `pid-N` /
`port-N` / `<productKey>-<version>` forms, no `displayName`,
`managedId`, or `lifecycleId` fields. The exposed name is the id
the JSON publishes, the id the CLI accepts as a positional
argument, and the id MCP tool calls use for `project_name`.

If a script needs a value that survives an IDE restart, use the
project's filesystem `path` (for projects) or query the
post-restart snapshot for the new exposed name (for backends).

`hash8` is informational only. **It is never parsed.** All reverse
lookups go through an in-memory `Map<exposed, route>` rebuilt from the
current discovery snapshot.

### Trust model

Markers under `~/.mcp-steroid/markers/`, managed-backend
descriptors under `~/.mcp-steroid/backends/`, and the IntelliJ
built-in HTTP server endpoints devrig probes for port-discovered
rows are all **trusted as inputs from the user's own user account**.
devrig does not validate marker contents against an external
authority; a marker pointing at an attacker-controlled `mcpUrl` would
route MCP calls to that endpoint.

This is acceptable because every input source lives inside the user's
own home directory (or, for port probes, inside `127.0.0.1`'s
loopback). Cross-user / privilege-boundary attack scenarios are out
of scope for this spec — if a different user can write to your
`~/.mcp-steroid/`, the attacker already controls your account.

A future hardening could add a marker signature / capability token.
That work is not part of this spec.

## Project naming

```
exposedProjectName = "${slug(originalProjectName)}-${hash8(canonicalProjectHome, idePid, bootHash)}"
```

`canonicalProjectHome = Path.of(projectPath).toRealPath()` — resolves
symlinks, so two paths that point at the same directory share a hash.
The same canonical path is what gets published as `projects[].path` in
the JSON; the spec does not preserve the unresolved symlink path.

`idePid` is the OS PID of the IntelliJ JVM that owns the project. It
appears in the hash AND is published as a data-model field in the
backend row so an operator can trace a project back to its OS
process. `bootHash` (next paragraph) makes the hash robust against
pid reuse; `idePid` makes the hash trivially explainable to a human
reading the JSON ("two rows differ → look at their pids").

`bootHash` comes from the marker and changes per plugin startup. It
guarantees the exposed name flips when the plugin restarts even if
the OS happens to reuse the same `idePid` for the new JVM.

Two `devrig` invocations separated by an IDE restart see different
hashes for the "same" project — the IDE generated a fresh `bootHash`,
and possibly took a different pid; both propagate into the hash.

### Invariants

1. Same canonical project path + same IDE pid + same `bootHash` ⇒ same
   `hash8`.
2. **Different `bootHash` (same canonical path, same pid) ⇒ different
   `hash8`.** Pids can be reused after a process exits; folding the
   plugin's per-startup `bootHash` into the hash makes the exposed
   name a true cross-startup identifier within a single `devrig`
   lifetime.
3. Different IDE pid (same project, same `bootHash`) ⇒ different
   `hash8`.
4. Different canonical project path (same pid, same `bootHash`) ⇒
   different `hash8`.
5. Two IDEs reporting the same `originalProjectName` produce two
   distinct `exposedProjectName`s.
6. Looking up an `exposedProjectName` in a fresh discovery snapshot
   uses the map directly. **No suffix parsing.**
7. A name that no longer maps fails with
   `ProjectRouteNotFoundException`:
   `"project_name '<name>' is no longer present; call steroid_list_projects to refresh"`
   See §"Live routing model" for the per-call selection contract.

Note: invariant 3 is theoretical inside a single devrig snapshot —
with ≥128 random bits in `bootHash`, two rows with the same
`bootHash` but different `idePid` is statistically impossible. The
invariant is stated for spec completeness and to give the test
contract a clean lemma; in practice `bootHash` carries the
uniqueness budget and `idePid` carries the debuggability budget.

### Implementation pointers (to be wired)

Spec is ahead of code. The bullets below name the call sites that
must be updated to match the contract; until they land, production
code still hashes `(canonicalProjectHome, idePid)` and composes the
exposed name without `slug()`.

- `DevrigProjectRoutingService.projectRoute` —
  `npx-kt/.../devrig/server/DevrigProjectRoutingService.kt:86-101`.
- `DevrigProjectRoutingService.projectRoute` must call
  `hash8(canonicalProjectHome, idePid, bootHash)` (currently
  `hash8(realHome, idePid)`).
- `DevrigProjectRoutingService.projectRoute` must compose the exposed
  name as `"${slug(originalProjectName)}-${hash8(...)}"` (currently
  `"${project.name}-$hash8"`).
- `ProjectRoute.projectPath` (publishing field) must carry the
  canonical (`toRealPath()`) path, not the raw `project.basePath`.
- `hash8` helper — same file, `:118-125`.
- The `hash8` helper must accept a third `bootHash: String` parameter
  and digest it after `realProjectHome\0idePid\0`.
- Exposed value carried in `ProjectRoute.exposedProjectName`.
- `bootHash` and `startedTime` must reach `ProjectRoute` from the
  marker (currently neither field exists in `PidMarker.kt`'s decoded
  model nor on the IDE-side identity API). The marker writer
  (`ServerUrlWriter` / `NpxBridgeService`) must publish both.
- Marker payload schema (`<pid>.mcp-steroid` JSON) gains
  `bootHash: String` and `startedTime: String` fields; both fields
  are also returned on the plugin's identity endpoint.
- The Kotlin implementation should carry an inline comment at the
  `hash8` call site explaining that `idePid` is intentionally
  included for debuggability even though `bootHash` already
  carries the uniqueness budget. (The comment lives in the code,
  not in the spec — but spec readers expect to find the rationale
  here so future implementers don't optimise the call to drop
  `idePid`.)
- Test contract: `DevrigProjectRoutingServiceTest` (and the inverse
  rewrites in `DevrigToolBridgeClientTest`).

## IDE (backend) naming

Applies the same pattern. Today only the locator label disambiguates
two IDEs of the same product+version; the display name is duplicated.
The spec promotes the hash suffix into the name itself so all backend
sources round-trip the same way.

```
exposedBackendName = "${slug(originalBackendName)}-${hash8(sourceKind, sourceKey)}"
```

| Source | `originalBackendName` | `sourceKind` | `sourceKey` |
|---|---|---|---|
| **Available download** | `"<displayName> <version>"`, where `displayName` is the human-readable product name (e.g. `"IntelliJ IDEA Community"`) rather than the slug (`"idea-community"`) | `"available"` | When `archiveSha256` is non-null: `"${productKey}\0${version}\0${archiveSha256}"`. When null: `"${productKey}\0${version}"` (fallback). |
| Marker (mcp-steroid plugin present) | `"<IdeInfo.name> <IdeInfo.version>"` | `"marker"` | `"${idePid}\0${bootHash}"` |
| Port (built-in HTTP server) | `productFullName ?? productName ?? "(unknown JetBrains IDE)"` | `"port"` | `port` |
| Managed (devrig-installed) | `"<productKey> <version>"` | `"managed"` | `info.id` |

When the products API publishes a checksum for the row, the exposed
`id` folds it in — so the id changes whenever the archive content
changes for the same `(productKey, version)` tuple. When the checksum
is unavailable, the id depends only on `productKey` and `version`;
consumers can detect this via the row's `archiveSha256Unknown` flag
and may choose to refuse the download or proceed with explicit user
opt-in.

`hash8` inputs use a null byte separator. Marker rows now have a
composite `sourceKey = "${idePid}\0${bootHash}"`, so the backend hash
is `hash8(sourceKind \0 sourceKey)`. Project hash and backend hash
both terminate at `hash8(input1 \0 input2 \0 …)`.

### Invariants

1. **Plugin restart ⇒ different `exposedBackendName`.** The plugin
   generates a fresh `bootHash` on every startup; folding it into the
   marker `sourceKey` guarantees a new exposed name even if the OS
   reuses the same pid.
2. Two IDEs of the same product+version running concurrently ⇒
   distinct `exposedBackendName`s.
3. An IDE moving from port-discovered to marker-discovered (i.e. the
   plugin gets installed) produces a **different** exposed name — the
   `sourceKind` changes. This is intentional; consumers should refresh.
4. Stale name lookup fails with `BackendRouteNotFoundException`:
   `"backend '<name>' is no longer present; call devrig backend (or steroid_list_projects) to refresh"`
   See §"Live routing model" for the per-call selection contract.

### Implementation pointers (to be wired)

- New helpers in `BackendIdentity.kt`:
  - `exposedBackendName(row: BackendRow): String`
  - `backendHash8(row: BackendRow): String`
- The source-specific backend label already returns
  `originalBackendName`. The exposed name is composed at render time.
- `backends[].id` is the exposed name. It is the only id this spec
  defines. Lifecycle commands consume the **same** exposed name as
  their positional argument — `devrig backend start <id>` /
  `devrig backend stop <id>` /
  `devrig backend provision <id>` accept the exposed name. The
  implementation parses the exposed name back to the underlying
  source row via the same in-memory map used for routing (no suffix
  parsing). Follow-up implementation task:
  `BackendCommand.kt`'s `isSupportedBackendLifecycleId` /
  `isSupportedProvisionTargetId` must resolve exposed names through
  the snapshot map and reject every non-exposed source identifier.
- Marker payload schema (`<pid>.mcp-steroid` JSON) gains
  `bootHash: String` and `startedTime: String` fields. Publish from
  `ij-plugin/.../server/SteroidsMcpServerStartupActivity.kt` (or the
  appropriate marker writer); both fields are also returned on the
  plugin's identity endpoint.

### `devrig backend --json` / `devrig project --json`

All JSON surfaces listed in this document are self-contained:
`devrig backend --json`, `devrig project --json`, every lifecycle
subcommand called with `--json`, and the devrig stdio MCP rows carry
the fields a script needs to drive a follow-up `devrig` command or an
MCP call. For row-based surfaces, `id` is the canonical reference (the
exposed name); source-specific identifiers (`pid`, `port`, `mcpUrl`)
are present when known; the `actions[]` array carries every applicable
follow-up command. The execution channel is `actions[].argv` (array,
shell-free). Scripts that exec via `actions[].command` (the derived
display string) MUST shell-quote the argument list themselves —
devrig does not guarantee `command` is safe to pass to `sh -c`. If a
row is missing an action, that action is not applicable to that row
in its current state.

**Per-row schema**

`backends[]`:

| Field | Meaning |
|---|---|
| `id` | Exposed backend name. Primary key for devrig backend JSON and CLI commands. CLI-only id; there is no backend identity in `steroid_list_projects`'s response shape today. |
| `originalName` | Raw label from the source (marker/port/managed). Display only. |
| `locator` | Human-readable "how do I reach this" hint. Never parsed. |
| `source` | `"marker"` / `"port"` / `"managed"`. |
| `managed` | `true` if this backend is devrig-managed. |
| `pluginInstalled` | `true` if the MCP Steroid plugin is present. A managed backend that is running with the plugin writes a marker and surfaces as `source = marker, managed = true` — it does **not** surface as `source = managed`. The `managed` source is reserved for managed backends that are installed but not running (no marker yet). |
| `reachable` | `true` if the backend currently answers HTTP / has a project snapshot. |
| `pid` | OS pid (marker / managed-running). Omitted otherwise. Published on marker rows (and on managed-running rows surfaced as markers). The same value is fed into the marker `sourceKey` and into the project hash, so operators can correlate JSON rows with OS processes without parsing the opaque suffix. |
| `bootHash` | Plugin-generated per-startup secret (marker rows only). Long base64url string. Folded into `id`'s hash for marker rows. |
| `startedTime` | ISO-8601 instant when the plugin generated `bootHash` (marker rows only). Part of the data model; not used in hash computation. |
| `port` | Built-in HTTP server port (port-discovered). Omitted otherwise. |
| `buildNumber` | IDE build number when known. |
| `installPath` | Filesystem path of the managed install (managed rows only). |
| `cachePath` | Filesystem path of the managed cache (managed rows only). |
| `mcpUrl` | MCP endpoint URL (marker only). |
| `plugin` | `{installed, id?, name?, version?}` — plugin identity when known. |
| `actions` | Every actionable follow-up command for this row in its current state, as `{id, label, argv, command}`.<br><br>`argv` — array of strings, the **canonical execution form**. Pass directly to `ProcessBuilder` / `posix_spawn`. No shell involved, no quoting required. Safe regardless of what characters survive `slug()`.<br><br>`command` — derived display string, `argv.joinToString(" ")` for human readability. **For humans only — do not `sh -c` this.** Spelling or metacharacters in `originalName` may render here unescaped.<br><br>`id`, `label` — unchanged.<br><br>Empty array means no actionable commands for this row. |

`projects[]`:

| Field | Meaning |
|---|---|
| `id` | Exposed project name. Primary key. Same string flows through `steroid_list_projects`'s `ProjectInfo.name` field and MCP tool calls' `project_name` parameter. |
| `originalName` | Raw `Project.name` from the running IntelliJ. Display only. |
| `path` | Canonical project base path (`Path.of(projectBasePath).toRealPath()`). Same value the hash computes over. Symlinks resolved. |
| `idePid` | OS PID of the IntelliJ JVM that owns this project row. Same value as `backends[].pid` for the row referenced by `backend`. Folded into the project hash for debuggability and as a redundancy check against `bootHash`. |
| `backend` | Foreign key to `backends[].id`. |
| `actions` | Reserved for project-level follow-up commands as `{id, label, argv, command}`. `argv` is the canonical execution form; `command` is display-only. Empty array means no actionable commands for the row in its current state. |

On Unix, `toRealPath()` resolves symlinks. On Windows, it additionally
resolves directory junctions and reparse points. Both produce a path
that uniquely identifies the on-disk location of the project home; the
spec does not promise any specific form (drive letter casing,
separator) — consumers compare canonical paths only with each other,
not against raw `Project.basePath`.

### Same project open in two IDEs

A single canonical project home can legitimately be open in two
running IntelliJ JVMs at the same time — e.g. one IDE for code
work, another for debugging or a different branch. The data model
supports this:

- The hash inputs (`canonicalProjectHome`, `idePid`, `bootHash`)
  produce **two distinct exposed names** because the two JVMs have
  different `idePid` AND different `bootHash`.
- The JSON publishes **two rows** in `projects[]`, each with the
  same `path` (the shared canonical home) but different `id`,
  different `idePid`, and different `backend` foreign keys.
- Each row routes independently — `steroid_execute_code` against
  one row's `id` runs in the corresponding IDE; against the other
  row's `id`, runs in the other.

This is the only place the spec allows duplicate `path` values in
`projects[]`. If two rows share `id` (and not just `path`) the
snapshot is malformed; see the routing-service invariant below.

### devrig stdio MCP — `steroid_list_projects`

`ProjectInfo.name` returned over the wire is always
`exposedProjectName`. Implementation: `DevrigListProjectsToolHandler`
in `StubMcpSteroidTools.kt:82-87`.

The same rule applies to `steroid_list_windows`. Both
`WindowInfo.projectName` and `WindowInfo.windowId` are rewritten by
`DevrigProjectRoutingService.rewriteWindow`:

- `projectName` becomes the exposed project name (or `null` when
  the window has no project context).
- `windowId` becomes `"<originalWindowId>-<projectHash8>"`, where
  `projectHash8` is the hash suffix of the owning project. A reverse
  map (`windowRoutes`) lets follow-up `steroid_input` /
  screenshot calls route back to the same IDE.

## Lifecycle subcommand JSON outputs

Each lifecycle command emits a single JSON document on stdout when
called with `--json`. All documents share a `tool` envelope
(`{name, version}`) and use the same naming/ID conventions as
`backend --json` / `project --json`. No alias fields exist.

### `devrig backend download` (list mode — no id)

`devrig backend download --json` (no positional id) lists IDEs
available for download.

```json
{
  "tool": { "name": "devrig", "version": "0.95.0" },
  "available": [
```

## Live routing model

See `docs/devrig-scanning-research.md` § "Decision" for the
rationale that led to on-demand scanning, and
`docs/PHILOSOPHY.md` § "Tenet 3 — devrig is stateless" for the
top-level invariant this section implements.

A single per-call routine (`rebuildSnapshot()`) constructs devrig's
live model on demand. It returns an immutable
`DiscoverySnapshot` with the same shape as the JSON data model
this spec defines for `devrig backend --json` /
`devrig project --json`.

`rebuildSnapshot()` runs three pieces in parallel:

- read every marker file under `~/.mcp-steroid/markers/` and
  parse each as JSON;
- scan the IntelliJ built-in HTTP server port ranges
  (`63342..63361`, `64342..64361`) with a short connect timeout
  (≤ 3 s per probe, parallel);
- enumerate managed-backend installs under
  `~/.mcp-steroid/backends/` and read each one's descriptor.

For every marker, the routine opens a one-shot HTTP POST to
`/npx/v1/projects/stream`, awaits the first `snapshot` envelope,
and closes. The IDE never sees a long-lived connection from
devrig.

`rebuildSnapshot()` is the **only** way to obtain a live model.
There is no `StateFlow`, no background `Job`, no per-IDE
persistent socket. devrig is stateless
across calls
(see `docs/PHILOSOPHY.md` § Tenet 3).

### Per-call routing

0. The caller MUST supply a non-empty `project_name` (for
   project-targeted calls) or backend `id` (for backend-targeted
   calls). Missing / empty / blank / null id fails immediately
   with `MissingProjectIdException` /
   `MissingBackendIdException` BEFORE `rebuildSnapshot()` is
   called.
1. The call's coroutine runs `rebuildSnapshot()`. The resulting
   snapshot is cached for the duration of the current call (so a
   call that needs the model twice doesn't double-scan).
2. The caller's id is looked up in the snapshot's
   `Map<exposed, route>`.
3. If the map has an entry, the call is dispatched to that
   backend.
4. If the map has no entry, the call fails with
   `ProjectRouteNotFoundException` /
   `BackendRouteNotFoundException` carrying the actionable "call
   …to refresh" message.

Per-call snapshot invariants:

- Every `projects[].id` is unique within one snapshot.
  Collisions are statistically impossible by construction
  (different `idePid`+`bootHash` ⇒ different hash).
- If the routing service nevertheless observes a collision on
  `id` while building the map, that snapshot is treated as
  malformed: the first row is kept, a warning is logged
  (channel pinned in §"Collision handling"),
  and routing continues.

### Collision handling

Every `projects[].id` is unique within one snapshot by construction
(different `idePid`+`bootHash` ⇒ different `hash8`). The same holds
for `backends[].id` across all four source kinds.

If `rebuildSnapshot()` nevertheless observes a collision on `id`
while building the `Map<exposed, route>`, the snapshot is treated as
**malformed**:

- The first occurrence (in iteration order) is kept.
- All subsequent rows with the same `id` are dropped from the map, but
  **published verbatim** in the JSON `projects[]` / `backends[]`
  array — so an operator running `devrig backend --json` can see the
  duplicate and diagnose.
- A warning is logged on the SLF4J logger
  `com.jonnyzzz.mcpSteroid.devrig.routing` at level `WARN`. The
  payload includes:
  - the duplicate exposed `id`,
  - the source kind of both rows (`"marker"` / `"port"` /
    `"managed"` / `"available"`),
  - the natural-source identifier of each row (pid + marker file path
    for marker rows; port for port rows; managed id for managed rows;
    productKey+version for available rows).
- No exception is thrown to the caller. Subsequent routing calls
  against the duplicated id resolve to the first occurrence; calls
  against any other id are unaffected.

This policy is intentionally not a security boundary — markers and
managed-backend descriptors live inside the user's own home directory,
so a malicious local user can already control the inputs. The policy
exists to give operators deterministic, observable behaviour when a
feed glitch or schema-drift produces colliding rows.

**Status:** to be wired. `DevrigProjectRoutingService.routes()` today
is silent last-writer-wins
(`linkedMapOf[route.exposedProjectName] = route`); the warning channel
above is the target shape. The backend-row JSON already implements
keep-first-and-warn in `BackendCommand.backendRowsWithStableIds`;
reuse that helper's logger when wiring projects.

### No implicit routing

devrig does not silently pick a backend when the id is omitted.
There is no "use the only IDE that happens to be running"
shortcut, no fuzzy match, no most-recently-used fallback.
Project-targeted MCP tool calls (`steroid_execute_code`,
`steroid_open_project`, `steroid_take_screenshot`,
`steroid_execute_feedback`, `steroid_list_windows`,
`steroid_fetch_resource`) require
`project_name`. Backend-targeted CLI commands
(`devrig backend start`, `stop`, `download`, `provision`) require
the positional id (the list-mode invocation with no id is a
separate command that emits the JSON catalogue, not implicit
routing).

If an MCP tool definition currently allows `project_name` to be
optional, the implementation MUST reject the call with
`MissingProjectIdException` before any snapshot work.

### In-flight calls are not re-routed

Once a call has been dispatched, it runs on the chosen backend
to completion. If the backend dies, becomes unreachable, or its
exposed id changes mid-call (e.g. plugin reload generates a new
`bootHash`, source-kind flip), the call surfaces whatever
transport error the bridge produces (HTTP status, channel
closed, SSE error event, etc.). **devrig does not retry,
mirror, or migrate in-flight calls.**

The next call repeats steps 0–4 against a fresh
`rebuildSnapshot()` and will succeed against the re-routed
backend if one exists.

### Snapshot latency

`rebuildSnapshot()` runs at MCP-call time. Expected wall-clock
on a typical dev box with 1-3 running IDEs:

| Step | Wall-clock |
|---|---|
| Markers dir `readdir` + JSON parse | < 5 ms |
| Port scan (40 TCP probes, parallel, fast-fail) | 50 – 150 ms |
| Per-IDE `/npx/v1/projects/stream` (parallel, one-shot) | 20 – 80 ms |
| Managed-backend enumeration | < 10 ms |
| **Total** | **75 – 250 ms** |

Worst case (one or more port probes hang against a half-open
socket): up to ≈ 3 s, bounded by the per-probe connect timeout.
The spec does not promise sub-200 ms latency; consumers that
need lower latency should batch multiple lookups inside a single
MCP call's lifetime (the per-call snapshot cache covers this).

## Non-goals

- **Not a stable cross-restart identifier (marker rows).** Marker
  rows' `sourceKey` includes `bootHash`, regenerated per plugin
  process. Restarting the IDE produces a fresh exposed name even if
  the OS reuses the same pid. For marker-stability, script against
  the project's filesystem `path`.
- **Port-discovered rows are stable across IDE restarts** that
  re-bind the same port — `sourceKey` is just the port number. If the
  IDE is restarted on a different port, the exposed name changes.
- **Managed-installed rows are stable across restarts** by design —
  `sourceKey` is `info.id`, which is the canonical install identifier
  and survives uninstall+reinstall as long as the install directory's
  `id` field is preserved.
- **Available-download rows** are stable as long as the products feed
  publishes the same `archiveSha256` for the `(productKey, version)`
  tuple; when the checksum is null, `id` is stable across runs
  against the same `(productKey, version)` only.
- **Not compatible with the in-IDE MCP Steroid endpoint.** That
  endpoint exposes raw `project.name`. devrig owns its own naming
  and consumers must pick one transport.
- **Not human-memorisable.** `hash8` is opaque. If you need to type
  a name, use shell completion against `devrig project --json` /
  `devrig backend --json`.
- **No alias IDs.** The spec defines exactly one id format
  (`<originalName>-<hash8>`). There is no migration path from a
  previous id format because none was ever published outside this
  spec.

  project home produce two distinct `projects[]` rows with the same
  `path`, different `id`, different `backend`. Each id routes to its
  own marker.

  the tool bridge; window-id round-trip via `rewriteWindow`.
- *(new)* `MarkerBootHashTest` —
  - Marker payload contains `bootHash` (≥ 22 base64url chars, no
    padding) and `startedTime` (ISO-8601) on every plugin startup.
  - Two consecutive plugin startups produce distinct `bootHash`
    values.
  - **Multiple `ServerUrlWriter` calls within one plugin process
    produce the SAME `bootHash`** — opening / closing projects, or
    triggering any other code path that refreshes the marker, does
    not change the published value.
  - The application service `McpSteroidBootHash` is instantiated at
    most once per plugin process. Run this test in an
    integration-test context where `replaceService` is NOT invoked on
    `McpSteroidBootHash` — a plain plugin integration test, or a
    `LightPlatformTestCase` with no service swap. Constructor
    instrumentation (`AtomicInteger.incrementAndGet()` + assert the
    counter is `1` after the full test run) is the safer formulation
    than relying on `service<...>` identity, because it surfaces
    accidental re-instantiation regardless of how the framework
    produces it.

  including the marker `sourceKey = ${idePid}\0${bootHash}`
