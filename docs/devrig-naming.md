# devrig naming ÔÇö projects & IDEs

This is the canonical spec for how the **devrig** binary names projects
and IDEs in every output it produces (CLI text, CLI JSON, devrig stdio
MCP).

## Scope

devrig is an **independent binary**. Its output is consumed by:

1. **Humans** reading `devrig backend` / `devrig project` text.
2. **Scripts** consuming `devrig backend --json` / `devrig project --json`.
3. **AI agents** calling `steroid_list_projects` / `steroid_list_windows`
   over the devrig stdio MCP server (`devrig mpc`).

All three surfaces share the names defined here. **They are not meant to
be valid against the in-IDE MCP Steroid server** ÔÇö that's a separate
endpoint with its own (legacy) naming. devrig is free to redefine.

## Vocabulary

| Term | Meaning |
|---|---|
| **original name** | The unmodified label as reported by the source. For a project: `Project.name` from the running IntelliJ. For an IDE: `"<IdeInfo.name> <IdeInfo.version>"` (marker), `productFullName` (port), or `"<productKey> <version>"` (managed). |
| **exposed name** | The canonical id string exposed by devrig. It is the only id this spec defines for a row. Always `<slug(originalName)>-<hash8>`, where `slug(s)` replaces every run of ASCII whitespace in `s` with a single `_`. The result is ASCII-printable and contains no spaces, so the exposed name is always a single shell word. |
| **hash8** | First 6 bytes of `SHA-256` over the canonical inputs, base64url-encoded without padding (8 chars, URL-safe). |
| **bootHash** | Long base64url string (Ôëą 22 chars, i.e. Ôëą 16 random bytes encoded without padding) generated **once per JVM start** by the MCP Steroid plugin and held in an `@Service(Service.Level.APP)`. Same value across the plugin process's entire lifetime. Written verbatim into every marker refresh. **Never regenerated by the marker writer** ÔÇö project open / close, plugin reload of unrelated state, and explicit marker rewrites all reuse the existing value. A fresh `bootHash` only appears when the IDE JVM restarts (or the plugin is uninstalled+reinstalled within the same JVM, which is treated as "plugin startup"). Published in the pid marker and on the plugin's identity API. |
| **archiveSha256** | Hex-encoded SHA-256 of the IDE archive that `devrig backend download <id>` would fetch. Always provided by the JetBrains products API for each `(productKey, version, os, arch)` tuple; the `available[]` rows publish it and the exposed `id` for each row folds it into the hash. |
| **startedTime** | ISO-8601 instant captured when the plugin generated its `bootHash`. Published in the same marker payload. Part of the JSON data model; **not** folded into `hash8` today. |
| **locator label** | The parenthesised "how do I reach this" hint, separate from the name. E.g. `(build IU-261ÔÇŽ, pid 24017)`. |

### Slug rule

`slug(s)` is the function that converts an `originalName` into the
shell-safe form used in exposed names. It MUST satisfy:

- Replace every maximal run of ASCII whitespace (space, tab, CR, LF)
  with a single `_`.
- Leave every other code point unchanged. In particular, do not
  lowercase, do not strip punctuation, do not normalise Unicode.
- Is idempotent: `slug(slug(s)) == slug(s)`.
- Total: defined for every input string including empty.

`originalName` continues to carry the unmodified label; only the
composed exposed name uses the slugged form. Consumers that need the
human-readable label use `originalName`; consumers that need a
routing key use `id`.

Examples:

| originalName | slug |
|---|---|
| `IntelliJ IDEA 2025.3.3` | `IntelliJ_IDEA_2025.3.3` |
| `IntelliJ IDEA Ultimate` | `IntelliJ_IDEA_Ultimate` |
| `idea-community 2025.3.3` | `idea-community_2025.3.3` |
| `myproject` | `myproject` |

### `bootHash` lifecycle

The IDE-side plugin owns `bootHash` generation. Implementation
contract (binding on every plugin release):

1. The plugin defines an application-level service:

   ```kotlin
   @Service(Service.Level.APP)
   class McpSteroidBootHash {
       val value: String = generateBootHash()  // call once, cached
       val startedTime: Instant = Instant.now()

       private fun generateBootHash(): String {
           val bytes = ByteArray(16)
           SecureRandom().nextBytes(bytes)
           return Base64.getUrlEncoder().withoutPadding().encodeToString(bytes)
       }
   }
   ```

2. `bootHash` is computed inside the service constructor ÔÇö i.e. on
   the **first** call to `service<McpSteroidBootHash>()`. Subsequent
   calls return the cached value.

3. Every marker writer call ÔÇö `ServerUrlWriter`, or any future
   writer ÔÇö reads `service<McpSteroidBootHash>().value` /
   `.startedTime` and inlines them into the marker payload. The
   writer MUST NOT call `generateBootHash` directly and MUST NOT
   construct a fresh `McpSteroidBootHash` instance.

4. Plugin hot-reload (IntelliJ's dynamic plugin reload during
   development) creates a fresh service instance and therefore a
   fresh `bootHash`. This is treated the same as a JVM restart ÔÇö
   exposed names flip.

5. `startedTime` is the moment the service was first instantiated.
   It is a published data-model field; not folded into the hash
   today.

The contract above is what callers see; the implementation may
differ in details (e.g. lazy `by lazy { ... }`, alternative
randomness source) as long as the **invariant holds**: within one
plugin process, every marker refresh carries the same `bootHash`,
and a fresh `bootHash` only appears when the plugin process
restarts (or hot-reload triggers re-instantiation).

### The only ID

Every consumer-visible reference to a project or a backend in any
devrig output uses the **exposed name** ÔÇö `<slug(originalName)>-<hash8>`
ÔÇö and nothing else. There are no alias IDs, no historical `pid-N` /
`port-N` / `<productKey>-<version>` forms, no `displayName`,
`managedId`, or `lifecycleId` fields. The exposed name is the id
the JSON publishes, the id the CLI accepts as a positional
argument, and the id MCP tool calls use for `project_name`.

If a script needs a value that survives an IDE restart, use the
project's filesystem `path` (for projects) or query the
post-restart snapshot for the new exposed name (for backends).

`hash8` is informational only. **It is never parsed.** All reverse
lookups go through an in-memory `Map<exposed, route>` rebuilt from the
current discovery snapshot.

## Project naming

```
exposedProjectName = "${slug(originalProjectName)}-${hash8(canonicalProjectHome, idePid, bootHash)}"
```

`canonicalProjectHome = Path.of(projectPath).toRealPath()` ÔÇö resolves
symlinks, so two paths that point at the same directory share a hash.
The same canonical path is what gets published as `projects[].path` in
the JSON; the spec does not preserve the unresolved symlink path.

`idePid` is the OS PID of the IntelliJ JVM that owns the project. It
appears in the hash AND is published as a data-model field in the
backend row so an operator can trace a project back to its OS
process. `bootHash` (next paragraph) makes the hash robust against
pid reuse; `idePid` makes the hash trivially explainable to a human
reading the JSON ("two rows differ Ôćĺ look at their pids").

`bootHash` comes from the marker and changes per plugin startup. It
guarantees the exposed name flips when the plugin restarts even if
the OS happens to reuse the same `idePid` for the new JVM.

### Invariants

1. Same canonical project path + same IDE pid + same `bootHash` Ôçĺ same
   `hash8`.
2. **Different `bootHash` (same canonical path, same pid) Ôçĺ different
   `hash8`.** Pids can be reused after a process exits; folding the
   plugin's per-startup `bootHash` into the hash makes the exposed
   name a true cross-startup identifier within a single `devrig`
   lifetime.
3. Different IDE pid (same project, same `bootHash`) Ôçĺ different
   `hash8`.
4. Different canonical project path (same pid, same `bootHash`) Ôçĺ
   different `hash8`.
5. Two IDEs reporting the same `originalProjectName` produce two
   distinct `exposedProjectName`s.
6. Looking up an `exposedProjectName` in a fresh discovery snapshot
   uses the map directly. **No suffix parsing.**
7. A name that no longer maps fails with
   `ProjectRouteNotFoundException`:
   `"project_name '<name>' is no longer present; call steroid_list_projects to refresh"`
   See ┬ž"Live routing model" for the per-call selection contract.

Note: invariant 3 is theoretical inside a single devrig snapshot ÔÇö
with Ôëą128 random bits in `bootHash`, two rows with the same
`bootHash` but different `idePid` is statistically impossible. The
invariant is stated for spec completeness and to give the test
contract a clean lemma; in practice `bootHash` carries the
uniqueness budget and `idePid` carries the debuggability budget.

### Implementation

- `DevrigProjectRoutingService.projectRoute` ÔÇö
  `npx-kt/.../devrig/server/DevrigProjectRoutingService.kt:86-101`.
- `hash8` helper ÔÇö same file, `:118-125`.
- Exposed value carried in `ProjectRoute.exposedProjectName`.
- Marker payload schema (`<pid>.mcp-steroid` JSON) gains
  `bootHash: String` and `startedTime: String` fields. Publish from
  `ij-plugin/.../server/SteroidsMcpServerStartupActivity.kt` (or the
  appropriate marker writer); both fields are also returned on the
  plugin's identity endpoint.
- The Kotlin implementation should carry an inline comment at the
  `hash8` call site explaining that `idePid` is intentionally
  included for debuggability even though `bootHash` already
  carries the uniqueness budget. (The comment lives in the code,
  not in the spec ÔÇö but spec readers expect to find the rationale
  here so future implementers don't optimise the call to drop
  `idePid`.)
- Test contract: `DevrigProjectRoutingServiceTest` (and the inverse
  rewrites in `DevrigToolBridgeClientTest`).

## IDE (backend) naming

Applies the same pattern. Today only the locator label disambiguates
two IDEs of the same product+version; the display name is duplicated.
The spec promotes the hash suffix into the name itself so all backend
sources round-trip the same way.

```
exposedBackendName = "${slug(originalBackendName)}-${hash8(sourceKind, sourceKey)}"
```

| Source | `originalBackendName` | `sourceKind` | `sourceKey` |
|---|---|---|---|
| **Available download** (catalogue row, IDE not yet installed) | `"<productKey> <version>"` | `"available"` | `"${productKey}\0${version}\0${archiveSha256}"` |
| Marker (mcp-steroid plugin present) | `"<IdeInfo.name> <IdeInfo.version>"` | `"marker"` | `"${idePid}\0${bootHash}"` |
| Port (built-in HTTP server) | `productFullName ?? productName ?? "(unknown JetBrains IDE)"` | `"port"` | `port` |
| Managed (devrig-installed) | `"<productKey> <version>"` | `"managed"` | `info.id` |

The `available` row's `sourceKey` includes `archiveSha256` so the
exposed `id` changes whenever the binary itself changes ÔÇö e.g. when
JetBrains re-publishes the same `(productKey, version)` tuple with a
different archive (rare but observed). Scripts that compared
run-to-run by `id` will see the change.

`hash8` inputs use a null byte separator. Marker rows now have a
composite `sourceKey = "${idePid}\0${bootHash}"`, so the backend hash
is `hash8(sourceKind \0 sourceKey)`. Project hash and backend hash
both terminate at `hash8(input1 \0 input2 \0 ÔÇŽ)`.

### Invariants

1. **Plugin restart Ôçĺ different `exposedBackendName`.** The plugin
   generates a fresh `bootHash` on every startup; folding it into the
   marker `sourceKey` guarantees a new exposed name even if the OS
   reuses the same pid.
2. Two IDEs of the same product+version running concurrently Ôçĺ
   distinct `exposedBackendName`s.
3. An IDE moving from port-discovered to marker-discovered (i.e. the
   plugin gets installed) produces a **different** exposed name ÔÇö the
   `sourceKind` changes. This is intentional; consumers should refresh.
4. Stale name lookup fails with `BackendRouteNotFoundException`:
   `"backend '<name>' is no longer present; call devrig backend (or steroid_list_projects) to refresh"`
   See ┬ž"Live routing model" for the per-call selection contract.

### Implementation pointers (to be wired)

- New helpers in `BackendIdentity.kt`:
  - `exposedBackendName(row: BackendRow): String`
  - `backendHash8(row: BackendRow): String`
- The source-specific backend label already returns
  `originalBackendName`. The exposed name is composed at render time.
- `backends[].id` is the exposed name. It is the only id this spec
  defines. Lifecycle commands consume the **same** exposed name as
  their positional argument ÔÇö `devrig backend start <id>` /
  `devrig backend stop <id>` /
  `devrig backend provision <id>` accept the exposed name. The
  implementation parses the exposed name back to the underlying
  source row via the same in-memory map used for routing (no suffix
  parsing). Follow-up implementation task:
  `BackendCommand.kt`'s `isSupportedBackendLifecycleId` /
  `isSupportedProvisionTargetId` must resolve exposed names through
  the snapshot map and reject every non-exposed source identifier.
- Marker payload schema (`<pid>.mcp-steroid` JSON) gains
  `bootHash: String` and `startedTime: String` fields. Publish from
  `ij-plugin/.../server/SteroidsMcpServerStartupActivity.kt` (or the
  appropriate marker writer); both fields are also returned on the
  plugin's identity endpoint.

**Per-row schema**

`backends[]`:

| Field | Meaning |
|---|---|
| `id` | Exposed backend name. Primary key for devrig backend JSON and CLI commands. CLI-only id; there is no backend identity in `steroid_list_projects`'s response shape today. |
| `originalName` | Raw label from the source (marker/port/managed). Display only. |
| `locator` | Human-readable "how do I reach this" hint. Never parsed. |
| `source` | `"marker"` / `"port"` / `"managed"`. |
| `managed` | `true` if this backend is devrig-managed. |
| `pluginInstalled` | `true` if the MCP Steroid plugin is present. A managed backend that is running with the plugin writes a marker and surfaces as `source = marker, managed = true` ÔÇö it does **not** surface as `source = managed`. The `managed` source is reserved for managed backends that are installed but not running (no marker yet). |
| `reachable` | `true` if the backend currently answers HTTP / has a project snapshot. |
| `pid` | OS pid (marker / managed-running). Omitted otherwise. Published on marker rows (and on managed-running rows surfaced as markers). The same value is fed into the marker `sourceKey` and into the project hash, so operators can correlate JSON rows with OS processes without parsing the opaque suffix. |
| `bootHash` | Plugin-generated per-startup secret (marker rows only). Long base64url string. Folded into `id`'s hash for marker rows. |
| `startedTime` | ISO-8601 instant when the plugin generated `bootHash` (marker rows only). Part of the data model; not used in hash computation. |
| `port` | Built-in HTTP server port (port-discovered). Omitted otherwise. |
| `buildNumber` | IDE build number when known. |
| `installPath` | Filesystem path of the managed install (managed rows only). |
| `cachePath` | Filesystem path of the managed cache (managed rows only). |
| `mcpUrl` | MCP endpoint URL (marker only). |
| `plugin` | `{installed, id?, name?, version?}` ÔÇö plugin identity when known. |
| `actions` | Every actionable follow-up command for this row in its current state, as `{id, label, command}`. `command` is a literal shell line ready to copy. Exposed ids never contain shell metacharacters ÔÇö the `slug` rule above guarantees they are a single argv token ÔÇö so `command` is safe to `sh -c` directly. Empty array means no actionable commands for this row in its current state. |

`projects[]`:

| Field | Meaning |
|---|---|
| `id` | Exposed project name. Primary key. Same string flows through `steroid_list_projects`'s `ProjectInfo.name` field and MCP tool calls' `project_name` parameter. |
| `originalName` | Raw `Project.name` from the running IntelliJ. Display only. |
| `path` | Canonical project base path (`Path.of(projectBasePath).toRealPath()`). Same value the hash computes over. Symlinks resolved. |
| `idePid` | OS PID of the IntelliJ JVM that owns this project row. Same value as `backends[].pid` for the row referenced by `backend`. Folded into the project hash for debuggability and as a redundancy check against `bootHash`. |
| `backend` | Foreign key to `backends[].id`. |
| `actions` | Reserved for project-level follow-up commands as `{id, label, command}`. Empty array means no actionable commands for the row in its current state. |

### Same project open in two IDEs

A single canonical project home can legitimately be open in two
running IntelliJ JVMs at the same time ÔÇö e.g. one IDE for code
work, another for debugging or a different branch. The data model
supports this:

- The hash inputs (`canonicalProjectHome`, `idePid`, `bootHash`)
  produce **two distinct exposed names** because the two JVMs have
  different `idePid` AND different `bootHash`.
- The JSON publishes **two rows** in `projects[]`, each with the
  same `path` (the shared canonical home) but different `id`,
  different `idePid`, and different `backend` foreign keys.
- Each row routes independently ÔÇö `steroid_execute_code` against
  one row's `id` runs in the corresponding IDE; against the other
  row's `id`, runs in the other.

This is the only place the spec allows duplicate `path` values in
`projects[]`. If two rows share `id` (and not just `path`) the
snapshot is malformed; see the routing-service invariant below.

### devrig stdio MCP ÔÇö `steroid_list_projects`

`ProjectInfo.name` returned over the wire is always
`exposedProjectName`. Implementation: `DevrigListProjectsToolHandler`
in `StubMcpSteroidTools.kt:82-87`.

The same rule applies to `steroid_list_windows`. Both
`WindowInfo.projectName` and `WindowInfo.windowId` are rewritten by
`DevrigProjectRoutingService.rewriteWindow`:

- `projectName` becomes the exposed project name (or `null` when
  the window has no project context).
- `windowId` becomes `"<originalWindowId>-<projectHash8>"`, where
  `projectHash8` is the hash suffix of the owning project. A reverse
  map (`windowRoutes`) lets follow-up `steroid_input` /
  screenshot calls route back to the same IDE.

## Lifecycle subcommand JSON outputs

Each lifecycle command emits a single JSON document on stdout when
called with `--json`. All documents share a `tool` envelope
(`{name, version}`) and use the same naming/ID conventions as
`backend --json` / `project --json`. No alias fields exist.

When called with an id, the JSON document reports the outcome of the
download:

```json

  "action": "download",
  "id": "idea-community_2025.3.3-1z8KqM03",

  "archiveSha256": "1a2b3c4d5e6f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8091a2b3c4d5e6f",

  "installPath": "/Users/me/.mcp-steroid/backends/idea-community-2025.3.3",
  "cachePath": "/Users/me/.cache/mcp-steroid/backends/idea-community-2025.3.3",
  "vmoptionsPath": "/Users/me/.mcp-steroid/backends/idea-community-2025.3.3/IntelliJ IDEA CE.vmoptions",
```

`id` is the **exposed backend name** of the newly-installed managed

so scripts can correlate the selected catalogue row with the action
result. `verifiedSha256` is `true` when devrig computed the actual
SHA-256 of the downloaded archive and it matched the published value.
On mismatch, action-mode JSON carries `verifiedSha256: false` and a
typed `archiveSha256Mismatch` error; the install is aborted and any
partial unpacked files are removed.

## Live routing model

A single in-process service holds devrig's **live model** ÔÇö a
snapshot of the same data shape this spec defines for the JSON
outputs. The model is maintained by passive discovery feeding into
the service:

- the marker scanner publishes / retracts marker rows as the
  per-pid marker files appear and disappear;
- the port scanner publishes / retracts port rows as the built-in
  HTTP servers come and go;
- the managed-backend service publishes / retracts managed rows as
  `devrig backend download / start / stop` complete.

### Per-call routing

Every incoming MCP call, and every CLI command that targets a
specific resource by id, **selects its target by consulting the
current snapshot**:

0. The caller must supply an explicit, non-empty id
   (`projects[].id` for project-targeted calls,
   `backends[].id` for backend-targeted calls). A missing or
   empty id fails immediately with
   `MissingProjectIdException` / `MissingBackendIdException`
   before any snapshot lookup is performed. The error message
   is: `"<project_name|backend> is required ÔÇö call devrig project / devrig backend (or steroid_list_projects) and use the published id."`
1. Look up the supplied id (`projects[].id` /
   `backends[].id`) in the snapshot's map.
2. If the map has an entry, dispatch the call to the corresponding
   backend.
3. If the map has no entry, fail with
   `ProjectRouteNotFoundException` (for a project id) or
   `BackendRouteNotFoundException` (for a backend id) carrying the
   "call ÔÇŽto refresh" message.
   - Snapshot invariant: every `projects[].id` is unique within one
     snapshot. The exposed-name hash inputs guarantee this by
     construction (different `idePid`+`bootHash` produce different
     hashes). If the routing service nevertheless observes a collision
     on `id` while rebuilding the map, that snapshot is treated as
     malformed: the service keeps the first occurrence, logs a warning,
     and continues. The collision case is never silently last-writer-
     wins, and never an exception thrown to the caller.

### No implicit routing

devrig does not silently pick a backend when the id is omitted.
There is no "use the only IDE that happens to be running" shortcut,
no fuzzy match, no most-recently-used fallback. The contract is:

- Project-targeted calls (`steroid_execute_code`,
  `steroid_apply_patch`, `steroid_open_project`,
  `steroid_take_screenshot`, `steroid_action_discovery`,
  `steroid_execute_feedback`, `steroid_list_windows`) require
  `project_name`.
- Backend-targeted CLI commands (`devrig backend start`,
  `devrig backend stop`, `devrig backend download`,
  `devrig backend provision`) require the positional id (the
  list-mode invocation with no id is a separate command that
  emits the JSON catalogue ÔÇö not implicit routing).

If an MCP tool definition currently allows `project_name` to be
optional, the implementation must reject the call with the
`MissingProjectIdException` shape above. The script-friendly
path is: call `steroid_list_projects` to get the published ids,
then supply one explicitly.

### In-flight calls are not re-routed

Once a call has been dispatched, it runs on the chosen backend to
completion. If the backend dies, becomes unreachable, or its
exposed id changes mid-call, the call surfaces whatever transport
error the bridge produces (HTTP status, channel closed, SSE error
event, etc.). **devrig does not retry, mirror, or migrate
in-flight calls.**

The next call repeats step 1ÔÇô3 against the then-current snapshot
and will succeed against a freshly-routed backend if one exists.

### Snapshot freshness is a property of the model, not the call

The freshness of the snapshot is whatever the passive scanners
publish ÔÇö typically sub-second for markers and ports, longer for
managed-backend state transitions. The spec does **not** mandate a
re-scan on every request. Callers who need a fresher view can call
`devrig backend` / `devrig project` (CLI) or
`steroid_list_projects` (MCP) which re-read the snapshot at the
moment of the call.

`hash8` is **not** a stable cross-restart identifier. Two `devrig`
runs separated by an IDE restart will see different hashes for the
"same" project; that's correct ÔÇö the IDE pid changed.

## Non-goals

- **Not a stable cross-restart identifier.** Exposed names fold in
  per-process inputs (`bootHash`, `idePid`, `port`, managed
  `info.id`) that change across IDE restarts. For cross-restart
  stability, script against the project's filesystem `path` or
  rediscover the exposed backend name from a fresh snapshot.
- **Not compatible with the in-IDE MCP Steroid endpoint.** That
  endpoint exposes raw `project.name`. devrig owns its own naming
  and consumers must pick one transport.
- **Not human-memorisable.** `hash8` is opaque. If you need to type
  a name, use shell completion against `devrig project --json` /
  `devrig backend --json`.
- **No alias IDs.** The spec defines exactly one id format
  (`<originalName>-<hash8>`). There is no migration path from a
  previous id format because none was ever published outside this
  spec.
